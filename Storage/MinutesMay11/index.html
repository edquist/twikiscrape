STORAGE TECHNICAL GROUP Minutes, 05/11/06

AGENDA

(1) Report on Upcoming SE Interoperability Meeting - Timur

(2) SE on OSG "How-To" - Brainstorming session - see below

(3) SE Support - revisit our model - led by Eileen

(4) Where do we go from here? (a) Local vs. WAN based storage (b) towards a roadmap

SE on OSG How-To: Eileen and I believe that an important missing item is a list of instructions for someone who wants to add a new SE to the OSG. What do they need to do?

We would like to have a brainstorming session on several issues which we need to discuss and then write-out:

(1) Requirements: What functionality does an OSG SE need to support?

(2) Validation: what tests need to be run? Is there an ACDC dashboard equivalent for SEs?

(3) Monitoring: who? what? how?

(4) Policy: how do you advertise/post your policies and capacities to other OSG users?

(5) Support: is there an OSG policy on support from providers? Is there general OSG support for new users of a new SE?

MEETING SUMMARY

Storage Interoperability Workshop

Timur: The main goal of workshop is to modify the SRM specification and implementations to satisfy needs of LHC experiments for storage.

Reps of experiments have stated that in its current form SRM does not satisfy their current needs - particularly storage permanence parameter (data retention - needs to be permanent). this is one of their concerns. goal will be to go over changes requested and finalize. reps from glue schema will also be present to discuss what changes this will require for them. There may be some clean up (deprecation) of functionality of glue scheme.

Eileen: what is output of meeting?

Timuir: particular protocol changes for SRM - new functionality, some old ones will be modified. also, implementation schedule will be agreed to. by major SRM storage systems. SRM milestones will be developed. Agreement between exps and SRM developers about what reqs and functionality will be. Reps from SEs: Castor, dCache, DPM, and DRM. and Expts.

Eileen: please come back and report highlights after workshop, is that alright? (Timur agrees)

Paul: I would encourage people to look at Michael Ernst's talk from today (link is to a pdf file), he goes into a bit more detail on all of this actually although he doesn't mention it directly. Also, Alex sent a report (again, a pdf file) to the storage list discussing interoperability tests which was very interesting and should be the subject of a future meeting.

OSG Storage Element Installation How To

What does it mean to put up an SE on the OSG? How do we do it?

David: certainly LIGO would be interested in a list of how to bootstrap a dCache facility. They are doing this and feel that the problems they encountered are generic.

Specifically: would love to have standard facilities that were supported somewhere in OSG that would allow us to test if our installation is working. In addition, he would like to be able to test at intermediate steps. right now he is trying to install, build several things: srmcp, DRM, trying to interoperate with other installations... need to be able to test individual pieces against standard, known things.

Surya: yes, I had the same issue.

Eileen: is storage doc in OSG any good? is it poorly organized, does it even exist?

Abhishek: we don't use OSG pages at all. we use internal mail lists, such as those maintained by Jon Baken for dCache.

Jon: just checked, and we already have SRM facilities that LIGO can use.

David: Yes, but we still need well known sites that are supported by community or by known entities that we could test on/against consistently. Currently I have been prevailing on people to provide these on a temporary basis, they are very busy, not their job to support them, etc. we (this group) should move this out to a supported infrastructure.

Surya: saves frustration, learning curve

Surya was asked to type up his notes on what it takes to bring up an SRM installation, and David will go over doc and add information and/or comment on his experience.

This will make a first draft.

Eileen: monitoring

Surya: monitoring at several different levels, but not clear where/what it is.

Paul: what about validation?

Eileen: OK, lets go back to that.

Paul: if you want to add a CE, you have to pass the ACDC tests and pass most of the tests.

David: I would just be happy with a command line "site verify" test, don't need the ACDC type thing initially. This is included in the VDT for CEs.

Paul: Alex has the equivalent for SEs (SRMs, at least) but not sure if it is in VDT. Mark Green talked about putting this into ACDC dashboard

Alex: there is a VDT tool, it is included there. The University of Iowa was using this.

Eileen: is there documentation for this SRM test? we could then add this to our How-To.

Back to Monitoring...

This would be important for new SEs.

David: There is an SRM tester Twiki page, which discusses the beginnings of a scheme that could be extended to do constant heartbeat "are you alive" testing. There are several SRMs that are already being monitored. There is only a small amount of info that is being tested, but this is a good skeleton on which a better system could be built.

Alex: yes, but there is also jClarens information that is being implemented in dCache as well as DRM. last I heard this was being implemented, and they were asking what information to put in there. (Dmitry at FNAL).

Paul: Alex, you joined meeting late [Note: he had warned us he would have to]. It has been requested to be able to install an SRM installation in pieces, with tests at each intermediate step to make sure that things have been installed/configured correctly.

Alex: yes it possible to do this, and it would be useful. should be able to do this.

Surya: and this is how I am doing this in installing an SRM implementation here.

Paul: what about a reference set of sites for this?
