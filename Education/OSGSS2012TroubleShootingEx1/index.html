Simple Troubleshooting

On Monday, you used the program 'simple'. We will use it again here. Copy the source file, simple.c to the exercise directory. 

[user@osg-ss-glidein touble_part1]$ cp /share/users/user/condor-test/simple.c ./

Next, we will compile the simple.c and create the executable simple 

[user@osg-ss-glidein touble_part1]$ gcc -static -o simple simple.c

How can we tell what type of file simple is? There is a special command for that: 

[user@osg-ss-glidein touble_part1]$ file simple
simple: ELF 64-bit LSB executable, AMD x86-64, version 1 (SYSV), for GNU/Linux 2.6.9, statically linked, for GNU/Linux 2.6.9, from 'mplete', not stripped

I highlighted an important bit above. It says 64 bit. simple is a 64 bit executable. Will this run on the 32bit machines? Lets find out...

We need the submit file for simple that you created Monday. We modified this submit file to target the 32bit machines. Copy this to a file named simple.submit: 

Universe   = vanilla
Executable = simple
Arguments  = 4 10
Log        = simple.log
Output     = simple.out
Error      = simple.error
should_transfer_files   = YES
when_to_transfer_output = ON_EXIT
Requirements = Arch == "INTEL"
Queue

Does the job complete?

Show Answer Hide Answer
No, the job doesn't complete. In simple.error, you can see the error: 
[user@osg-ss-glidein touble_part1]$ cat simple.error
/myhome1/condor/execute/dir_9935/glide_D10001/condor_job_wrapper.sh: line 8: /myhome1/condor/execute/dir_9935/glide_D10001/execute/dir_14214/condor_exec.exe: cannot execute binary file
/myhome1/condor/execute/dir_9935/glide_D10001/condor_job_wrapper.sh: line 8: /myhome1/condor/execute/dir_9935/glide_D10001/execute/dir_14214/condor_exec.exe: Success

Ignore the Success, clearly the earlier error (cannot execute binary file) means the job didn't run.

So, what can we do to solve this problem?

  1. We can recompile simple for 32 bit.
  2. We can target only 64 bit machines.

Recompiling for 32bit machines is easy. Just need to send a new command argument to gcc when you are compiling the executable: 

[user@osg-ss-glidein touble_part1]$ gcc -static -m32 -o simple simple.c

Retargeting the job for only 64 bit machines requires chaning the =requirements statement in the submit file: 

Requirements = Arch == "X86_64"

What are the advantges of either approach? Does the job complete?

Show Answer Hide Answer
When you recompile for 32bit machines, that means you can run on both 32bit and 64bit machines (as seen in the presentation). This widens the resources available to run your processing.

Changing the requirements to target 64bit machines helps if it is difficult to rebuild the executable (for example, if you do not have the source). Also, if the application needs 64 bit for some reason (over 4GB of ram required?).

Bigger Troubleshooting

Lets use the shell script we used on Monday as well. Copy script into a file named simple.sh. 
#!/bin/sh

if [ $# -ne 2 ]; then
    echo "Usage: simple.sh  "
    exit 1
fi

echo "Thinking really hard for $1 seconds.."
sleep $1
answer=$(( $2 * 2 ))
echo "We calculated $answer."
exit 0

Next, modify simple.submit to run this executable rather than the compiled simple. When we submit the job, everything works great. But lets make it work... not great. Lets make it fail.. you know... for science!

Lets modify simple.submit to send an odd argument. Modify the arguments line, replacing the second number with a word (for example, blah). Submit the job to condor (condor_submit simple.submit). What is the output after the job has completed?

Hum... something is not right. Lets see what is going on in the shell script. At the top of the shell script, modify the line #!/bin/sh to add a -x. Therefore, the top line should be: #!/bin/sh -x. Submit the job again. After the job has completed, look in the simple.error.

Show example simple.err Hide example simple.err 
[user@osg-ss-glidein touble_part1]$ cat simple.error 
+ '[' 2 -ne 2 ']'
+ echo 'Thinking really hard for 4 seconds..'
+ sleep 4
+ answer=0
+ echo 'We calculated 0.'
+ exit 0

Odd, we didn't send it 0. We sent it blah, but it interpreted it as 0. This is a quirk of bash shell scripts, if it doesn't understand a number (ie, if it's letters), then intrepret it as 0.

Can you imagine scenerios where this would be helpful? Debugging shell scripts is very common in grid computing, since we have so many wrapper scripts.

Adding condor_ssh_to_job

Fix the simple.submit above, replacing blah with 10. Also, make the shell script sleep longer, change the first number in the arguments line to be 20. This time, we want to watch the job execute while it happens.

Submit the simple.submit. Watch it's execution in the queue using condor_q. When the job begins running, ie you see a R in the job status, issue the command: 

condor_ssh_to_job 10.0

Replacing 10.0 with your actual job id from condor_q. After issueing this command, you should be logged into the remote machine. You can see the files and job running on the execute machine.

Looking around at the files on the remote worker node, can you answer some simple questions:

  1. What files do you have in the directory.
  2. Can you guess what these files will be named once they are transferred back?
  3. What directory are you in? Is it where you expected?

Once you are done looking around on the worker node, type logout in the screen.
