Run a BLAST example - The right way

You should have experienced some problems running the BLAST jobs without any changes.

How would you fix the problem?

Show solution 1. Hide solution 1.
The glideinWMS policy allows us to only run on sites that have BLAST pre-installed; it was mentioned in the glideinWMS policy document.
Just add 
+REQUIRE_BLAST = True
to the job submission file.

One thing it is not in that document, is that different sites may have BLAST installed in a different location. For this reason, the glideins also advertise 

BLAST_PATH
in the environment. I.e. the BLST executable is now in 
$BLAST_PATH/bin/blastp
. Change your wrapper script accordingly.

Moreover, we know BLAST is installed locally, but the default glideinWMS policy prevents us to run locally.
Change this by adding also 

+OFFSITE_ONLY = False
to the job submission file.

Try them out (say, a cluster of 5), see where they run and what results you get back.


Show solution 2. Hide solution 2.
Instead of restricting ourselves to a subset of sites, let's instead transfer the necessary files to the worker node.

We need two pieces:

  • The binary itself.
    Since blast is statically linked, just transfer the file itself and execute it.
  • The database.
    Here we need more than a single file; the database is composed of many files; in case of the yeast DB, it is yeast.aa*
    The best way to do it, is to tar them up and untar them once on the worker node.

Can you change the submit file and the wrapper script accordingly?

# create tar file
m=$PWD
cd /share/blast/data/; tar -czf $m/yeast.tgz yeast.aa*
cd $m


Universe   = vanilla

transfer_input_files=query1,/share/blast/bin/blastp,yeast.tgz
should_transfer_files   = YES
when_to_transfer_output = ON_EXIT
Queue 5


#!/bin/sh

if [ $# -ne 2 ]; then
    echo "Usage: run-blast  "
    exit 1
fi

chmod a+x blastp
tar -xzf yeast.tgz

# Run the query
./blastp -db yeast.aa -query $1  > $2
