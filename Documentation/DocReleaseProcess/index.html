Key Policies Affecting the Documentation Release Process

  1. OSG releases are numbered r.n.p (e.g. 1.2.5, or 3.0.5).
  2. Major releases are denoted by a change in "r" or "n" which is always an even number. Odd numbers of "n" are used for integration testing but never an official release to the consortium.
  3. OSG's policy is to support the current and previous major release (e.g. 1.2.x & 3.0.x).
  4. Point releases are denoted by a change in "p" (e.g. 1.2.4 to 1.2.5). Those can be frequent (a month apart) and often contain significant new functionality desired by OSG members. Typically, these do not have a major affect on documentation, but it can have some affect on particular documents.
  5. For RPM releases there is a known correspondence between the OSG version and version of the RPM packages included in it

Documentation Release Process

Although the following represents the standard process, judgment of the Trash/Trash/Integration Team will determine how a particular case will be handled.

Major Releases

The process used on Major Releases is as follows, where Release4 is the new release, Release3 is the current production release and Release1.2 is the previous production release:
  1. The Release4 subweb is created and and a migration table for all the topics in the Release3 documentation tree will be created (including WebLeftBar, which will be the first document to be copied in the new Release4)
  2. After a quick call each document in Release3 will be either:
    • Discarded because it is no longer relevant (or significant)
    • Copied to Release4 as released document (RELEASE_READY) if the document needed no changes or quick changes (like fixing links to Release3 to point to Release4)
    • Copied to Release4 as document requiring editing (not RELEASE_READY) if the editor or reviewer changed or the document needed some changes
  3. The owner, tester and reviewer of the copy in Release4 are the same of the original document in Release3. If a person is no longer available, a new OSG member is assigned and the document is marked RELEASE_READY=NO
  4. The internal links between pages in the new Release4 web should be relative links whenever possible and absolute links can be are changed using unix tools in a supervised manner (represented by the bottom right red link in the diagram below).
  5. The Release4.WebLeftBar links can be modified by manually editing Release3 to Release4 or using unix tools in a supervised manner (represented by the bottom left red link in the diagram below). Again, relative links are preferred when possible.
  6. The Documentation Team and Software Team modify the documents in Release4 as needed. Each editor is responsible for its own documents. The Documentation Team and Software Team may change a document and notify the owner.
  7. Upon formal release, the links in the Documentation Web are changed from Release3 to Release4 using unix tools in a supervised manner (represented by the 2 vertical red links in the center of the diagram below). References to Release3 have to be added whenever there are references to Release1.2 supported releases different from the current production one
  8. The Documentation.WebLeftBar is edited to modify Release3 links to Release4 and to update the previous and current release links (represented by the 2 red links from Doc LeftBar box in the diagram below). References to Release3 have to be added whenever there are references to Release1.2 supported releases different from the current production one. The "Current Release" subweb of Documentation will always point to the current production release. Navigation into the "Current" release documentation will always default to the "Current Release" subweb. A pointer on the Documentation.WebHome navigation page will direct a user to the documentation tree for the previous release.
  9. Release1.2 (and previous releases) become unsupported. For each one, e.g Release1.2 the following steps are necessary:
    1. Release1.2 web is moved as subweb to ArchivedDocumentation (Trash/Trash/ArchivedDocumentation/OSG/OSG120)
    2. All references to Release1.2 are removed from the Documentation web. Exceptions are possible for documents about OSG history, e.g. mentioning the evolution of OSG monitoring. Links in those documents should be updated to point to the new Trash/Trash/ArchivedDocumentation/OSG/OSG120 web.
    3. Links to Release1.2 are added to topics or Navigation Pages mentioning archived releases
    4. Release3 and Release4 webs are checked against references to Release1.2 (there should be already no reference to Release1.2). Any reference found is removed.
  10. Changes that are made to improve documents that are independent of the new release will have to be made in the both subwebs just as they are between the current and previous release.

Point Releases

  1. For point releases with few changes, the documentation will be modified with notes so that the documents will serve for all point releases within a major release.
  2. For point releases with more extensive changes which should always be concentrated in a particular area, the (few) documents in that area (i.e. the set of documents with the more extensive changes) will be split into a version for the previous release and a version for the new release. The process will be:
    • The original document(s) (e.g. DocName) will be copied with the name changed by appending a number representing the release r.n.p (e.g. 3.0.5 becomes 305 appended to the TWiki word, as in DocName305). Another copy will be modified for the new point release and named by appending its release r.n.p+1 (e.g. 3.0.6 becomes DocName306). The original document with the original TWiki name (e.g. DocName) will be changed to describe the two documents and releases to which they they apply. For the next major release, the latest version r.n.p+1 (e.g. DocName306) can be copied back to the original (e.g. DocName) and the previous version (e.g. DocName305) will be deleted so that there will again be only one version of the document.
    • Note that for RPM distributions the RPM version of the packages involved may need to be mentioned

Temporary procedures for the ReleaseDocumentation web

The new Twiki structure was defined with OSG Release3. The documentation for Release1.2 resides in the ReleaseDocumentation web outside the Documentation tree. For the purpose of this document ReleaseDocuemtation should be considered equivalent to a Documentation subweb, e.g. Documentation/Release1.2. Here some exceptions that will remain in effect until OSG Release 1.2 is supported:
  • The ReleaseDocumentation web will remain where it is, as root web until Release 1.2 is supported
  • The ReleaseDocuemtation.WebLeftBar will not be edited except for adding links to the main Documentaiton hub and a notification that the release is no longer the current production release
  • When it is archived, it will move to Trash/Trash/ArchivedDocumentation/OSG/OSG120 and all the links will be updated
