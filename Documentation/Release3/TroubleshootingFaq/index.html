About this Document

This document is for grid users. It provides answers to frequently asked questions regarding the submission of grid jobs on the Open Science Grid.

Frequently Asked Questions

Condor-G: Log File Locations?

  1. Check Condor job description file for the log file of the job.
  2. Check Grid Manager log file on the submission host:
    $> cat $CONDOR_CONFIG | grep -P '^GRIDMANAGER_LOG'
GRIDMANAGER_LOG = $(LOG)/GridmanagerLog.$(USERNAME)
$> cat $CONDOR_CONFIG | grep -P '^LOG'
LOG             = $(LOCAL_DIR)/log
$> cat $CONDOR_CONFIG | grep -P '^LOCAL_DIR'
LOCAL_DIR       = /var/condor
  3. Check Condor log files on the submission host:
    $> cat $CONDOR_CONFIG  | grep -v MAX | grep -P '^\w+_LOG\s*\='
COLLECTOR_LOG            = $(LOG)/CollectorLog
KBDD_LOG                 = $(LOG)/KbdLog
MASTER_LOG               = $(LOG)/MasterLog

Condor-G: How to find the 'hold' code?

  1. You can find the error code in the log file specified in the condor submit file. An error entry will look something like this:
            018 (035.000.000) 04/10 12:01:04 Globus job submission failed!
            Reason: 5 the executable does not exist

        012 (035.000.000) 04/10 12:01:04 Job was held.
            Globus error 5: the executable does not exist
            Code 2 Subcode 5
  2. To find all jobs that are being held by Condor use the condor_q command:
    [user@ce /opt/osg-1.2]$ condor_q -hold
-- Submitter: ce.opensciencegrid.org : <IP:PORT> : ce.opensciencegrid.org
 ID      OWNER           HELD_SINCE HOLD_REASON                   
4989741.0   user          5/23 17:08 Globus error 47: the gatekeeper failed to r
...
    Take note of the ID = 4989741.0 which can be used to retrieve more details on the problem:
    [user@ce /opt/osg-1.2]$ condor_q -l 4989741.0 | grep HoldReason
LastHoldReason = "Globus error 47: the gatekeeper failed to run the job manager"
LastHoldReasonCode = 2
LastHoldReasonSubCode = 47
HoldReason = "Globus error 47: the gatekeeper failed to run the job manager"
HoldReasonCode = 2
HoldReasonSubCode = 47
    In the example above the major error code 2 refers to a Globus problem on the remote site. The minor error code 47 means that the gatekeeper failed to run the job manager. For a list of Globus Errors, see Globus Errors. For a list of Condor hold codes by reason, see Condor Errors.

Condor-G: How to turn on full debugging?

Make the following change to your $CONDOR_CONFIG: 

GRIDMANAGER_LOG = D_FULLDEBUG

and execute condor_reconfig: 

$> condor_reconfig
Sent "Reconfig" command to local master

Condor-G: How to run the Grid Manager manually?

  1. Find the Condor location on the remote site:
    $> globus-job-run ce.opensciencegrid.org/jobmanager-fork /bin/bash -c "echo $CONDOR_CONFIG"
/usr/local/condor-7.4.0/etc/condor_config
  2. Start the grid_monitor.sh script on the remote site:
    $> globusrun -s -r ce.opensciencegrid.org/jobmanager-fork '&(executable= $(GLOBUSRUN_GASS_URL)/usr/local/condor-7.4.0/sbin/grid_monitor.sh) (arguments="--dest-url="#$(GLOBUSRUN_GASS_URL)#"/tmp/job_status")'
2011-08-04 11:56:37 OK: 
2011-08-04 11:56:37 INFO: Forced agent start
2011-08-04 11:56:37 INFO: Starting grid_manager_monitor_agent
2011-08-04 11:56:37 INFO: Started grid_manager_monitor_agent as /tmp/grid_manager_monitor_agent.ligo.29376.1000, pid 29377
2011-08-04 11:56:37 INFO: grid_manager_monitor_agent already running.
2011-08-04 11:57:37 OK:
  3. Check file /tmp/job_status as specified above:
    $> cat /tmp/job_status
1312484320 1312484321
https://ce.opensciencegrid.org:32780/23541/1311048510/          8
https://ce.opensciencegrid.org:32809/13208/1310864843/          8
...
GRIDMONEOF
    The file content will be refreshed every minute. The first column on each line is the GRAM JobID. The second column is the GRAM job status (1=pending, 2=active, 8=done). All of the GRAM JobIDs in the condor_q output associated with the resource (ce.opensciencegrid.org) should appear in /tmp/job_status.

Condor-G: What are the firewall requirements?

If you find that a Globus command succeeds while an equivalent condor-G job to the same site fails, it is likely that the problem is being caused by a firewall blocking the network ports. Condor-G needs more network port open since Globus on the remote site needs to contact Condor-G, and a firewall that blocks incoming connections prevents that connection from being made (It is possible, though unlikely, that the remote site is blocking the outgoing connection). Note that you need a port range with at least three ports per user (Distinguished Name) per site for use by Condor-G.

  1. In the file $CONDOR_CONFIG set the following values:
    HIGHPORT = <high port>
LOWPORT  = <low port>
  2. Execute condor_reconfig:
    $> condor_reconfig
Sent "Reconfig" command to local master
  3. Allow UDP connections in the specified range.

Further information on Compute Element firewall requirements can be found Firewall Information.

Condor-G: Why is the GridManager not running?

Running the Condor-G GridMonitor is considered a best practices for running a large batch of jobs on the Open Science Grid. The easiest way to determine if the GridMonitor is running from the client side is to check the GridManager log file: 

$> cat $CONDOR_CONFIG | grep -P '^GRIDMANAGER_LOG'
GRIDMANAGER_LOG = $(LOG)/GridmanagerLog.$(USERNAME)
$> cat $CONDOR_CONFIG | grep -P '^LOG'
LOG             = $(LOCAL_DIR)/log
$> cat $CONDOR_CONFIG | grep -P '^LOCAL_DIR'
LOCAL_DIR       = /var/condor

Jobs that stream stdout and/or stderr will not use the GridManager. If streaming of stdout and/or stderr is not required, include the following lines in the Condor job description: 

stream_output = false
stream_error  = false

Make sure that the Condor variables GRID_MONITOR and ENABLE_GRID_MONITOR are set correctly in $CONDOR_CONFIG: 

$> cat $CONDOR_CONFIG | grep GRID_MONITOR
GRID_MONITOR        = $(SBIN)/grid_monitor.sh
ENABLE_GRID_MONITOR = TRUE

Query the running Condor directly for the variables above: 

$> condor_config_val grid_monitor
/usr/local/condor-7.4.0/sbin/grid_monitor.sh
$> condor_config_val enable_grid_monitor
TRUE

GRAM: How to save the jobmanager logs?

GRAM can be configured on the Compute Element by editing $GLOBUS_LOCATION/etc/globus-job-manager.conf and changing the line -save-logfile: 

[root@ce /opt/osg-1.2]$ cat $GLOBUS_LOCATION/etc/globus-job-manager.conf
	-home "/opt/osg-1.0.5/globus"
	-globus-gatekeeper-host osg-ldg.ligo.caltech.edu
	-globus-gatekeeper-port 2119
	-globus-gatekeeper-subject "/DC=org/DC=doegrids/OU=Services/CN=osg-ldg.ligo.caltech.edu"
	-globus-host-cputype x86_64
	-globus-host-manufacturer unknown
	-globus-host-osname Linux
	-globus-host-osversion 2.6.18-128.7.1.el5xen
	-globus-toolkit-version 4.0.8
	-save-logfile always
	-state-file-dir /opt/osg-1.0.5/globus/tmp/gram_job_state
	-machine-type unknown

GRAM: 'Host key verification failed'?

This error may occur when a job is submitted using globus-job-run to a Compute Element that uses the Portable Batch System. The submission error may look like: 

$> globus-job-run ce.opensciencegrid.org/jobmanager-pbs /bin/date
Host key verification failed.
/var/spool/torque/mom_priv/jobs/1073.everes.SC: line 87: [: too many arguments

This problem often occurs if the Compute Element has not been configured to use ssh key-pairs internally, which would allow all the parts of the multiple job to be launched by the gatekeeper. The RSL jobtype variable knows three different arguments: single, multiple (default) and MPI. To fix this problem you need to add an RSL statement '(jobtype=single)' to your RSL job description: 

$> globus-job-run ce.opensciencegrid.org/jobmanager-pbs -x '(jobtype=single)' /bin/date

See More PBS Batch System Hints.

GRAM-WS: 'Certificate not yet valid'?

The grid certificate used to submit the job is likely valid. However Globus GRAM-WS is sensitive to clock-skew between the client and remote side. The solution is to synchronize the client clocks with NTP (This can also be caused from the clock skew on server side, though it is unlikely).

'Can't find vomses'?

When the voms-proxy-init command is run, the error 'Cannot find file or dir: =/home/condor/execute/dir_14135/userdir/glite/etc/vomses=' occurs.

The error message does not really indicate an error. It is an artifact of the software build used in the Virtual Data Toolkit. Refer to the release documentation for more details.
