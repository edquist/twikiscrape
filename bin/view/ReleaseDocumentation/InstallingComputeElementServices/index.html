Installing Compute Element Services


Introduction

This guide covers the installation of the OSG Compute Element (CE). The CE is the server software that allows grid jobs to enter and run on your cluster. If you are unsure of what this does, read the OSG services overview.

The install is mostly a straightfoward process; in our experience, most problems occur due to skipping steps in the pre-install checklist and in configuring the software (as the configuration step requires lots of precise information to be filled in).

Generally, this page should take 10 - 60 minutes to follow; it may be that the preparation page takes longer to follow. If you find your install is taking more than an hour, refer to the contacts page to find out where you can get help.

Applicable Versions
This document focuses on the latest release, OSG-1.2.28 . However, we attempt to make notes where there are major differences between OSG 1.2.28 and prior versions.

Before you Install

This document is not standalone. New sites should definitely run through the Site Planning Guide. New CEs should read through the Preparing the Compute Element page.

You MUST read the pre-install checklist; this covers setting important environment variables critical to the install and the creation of user accounts.

on Common pitfalls:

  • Skipping the pre-installation checklist.
  • If you've just installed the workernode client on your CE, log out and log back in again so that your PATH doesn't become confused.
  • If you have an existing CE installed and running, you must do a complete shutdown and confirm that all processes associated with it have been stopped. Refer to Shutting Down for details. Remember you are not shutting down this version, so follow the instructions in the documentation for the previously installed version. After shutting down, make sure to log out and log back in again so that your PATH doesn't become confused in the subsequent commands.
  • Finally, make sure there are no non-standard versions of perl, python, tcsh, or bash in your $PATH variable.

Install the Compute Element

Now you're ready to start installing the Compute Element.

Create a directory such as /opt/osg-1.2.32 and create a sym link /opt/osg that points to this directory and then go into this directory: cd /opt/osg-1.2.32. Please do not install in a directory like /opt or /usr, because the installation will create many subdirectories in its top-level directory. From here forward this directory is known as the $VDT_LOCATION; after you setup the OSG environment, this variable will be in your environment.

The installation described here is done as root even though services will not necessarily run as root.

As described in the preparation document, you should already have pacman installed and the pacman command in your environment. For example, OSG 1.2.28 requires pacman 3.28, so check to see that you have the correct version of pacman in your path: 

which pacman
pacman -version

Run the following to install the CE software: 

pacman -get http://software.grid.iu.edu/osg-1.2:ce

Pacman will ask you if you trust of the caches that deliver the software.
Please answer y (yes) so that the software can be retrieved. See the Pacman section of this guide if you encounter an 'unsupported' platform message.

NOTE
This will take between 5 and 30 minutes to complete, depending upon the system and network connection. During this time you may open a second terminal and watch the progress by monitoring the $VDT_LOCATION/vdt-install.log file.

NOTE
On 64-bit systems you may see the following informational messages. These components have never been available for 64-bit systems, but this is not a problem because in practice they are not used in OSG. 
INFO: The Globus-Base-Info-Server package is not supported on this platform
INFO: The Globus-Base-Info-Client package is not supported on this platform

After the initial question about caches, you should not be asked any other questions during the installation process. The installation should complete with the following message: 

Downloading [vo-package-6.tar.gz] from [http://software.grid.iu.edu/osg-1.2/tarballs]...  
Untarring [vo-package-6.tar.gz]...  
Existing /usr/local/vdt-1.10.0/edg/etc/edg-mkgridmap.conf moved to /usr/local/vdt-1.10.0/edg/etc/edg-mkgridmap.conf.orig 
Merge it manually with the new /usr/local/vdt-1.10.0/edg/etc/edg-mkgridmap.conf if you had a special edg-mkgridmap.conf 
Pacman Installation of OSG-1.2.32 Complete
In the latest release the install may complete with the following message if you set OLD_VDT_LOCATION to preserve previous configuration. 
All prerequisite checks are satisfied.
Your CA certificates have been setup automatically using information from
OLD_VDT_LOCATION.  Verify the link at VDT_LOCATION/globus/TRUSTED_CA
                                                                              
Pacman Installation of OSG-1.2.32 Complete

WARNING: Do not continue if the pacman command ended in an error.

Setup the OSG environment

You should now source the OSG environment. Change your directory to $VDT_LOCATION (the location where you installed the software) and run, 

 source setup.sh
or 
 source setup.csh
as appropriate for your shell.

If the setup.sh or setup.csh files do not exist, your install in the previous section failed; see $VDT_LOCATION/vdt-install.log for more information. Do not proceed in this case.

Install job managers

You need to install an extra package to interface with your existing Condor, PBS, LSF, or SGE installation.

NOTE
The PBS and Condor batch systems require special consideration:
  • For PBS, make sure your PBS binaries (qsub, qstat, qdel) are in the $PATH prior to installation. Otherwise the install will fail silently with error reported to vdt-install.log and jobmanager-pbs will not be created. If this is done, do a pacman -remove http://software.grid.iu.edu/osg-1.2:Globus-PBS-Setup, insert the binaries into your PATH and install again.
  • For Condor, have the following environment variables set (otherwise the package will install a complete version of Condor):
    • VDTSETUP_CONDOR_LOCATION: the location of your Condor installation (e.g. /opt/condor). The Condor bin/, sbin/, etc/, lib/ ... directories should be directly under this location.
    • VDTSETUP_CONDOR_CONFIG: The location of your Condor configuration file. Defaults to =${VDTSETUP_CONDOR_LOCATION}/etc/condor_config.

After you ensure that that the command-line utilities for your batch system are in your path,, run the command below relevant to your batch system: 

pacman -get http://software.grid.iu.edu/osg-1.2:Globus-Condor-Setup 
pacman -get http://software.grid.iu.edu/osg-1.2:Globus-PBS-Setup
pacman -get http://software.grid.iu.edu/osg-1.2:Globus-LSF-Setup  
pacman -get http://software.grid.iu.edu/osg-1.2:Globus-SGE-Setup

These packages will install hooks for Globus pre-WS, Globus web-services (WS), and Gratia accounting probes.

NOTE
Globus-Condor-Setup only - The $VDT_LOCATION/globus/lib/perl/Globus/GRAM/JobManager/condor.pm will by default insert a requirement that all jobs run on nodes of the same architecture as the gatekeeper. i.e. if your gatekeeper is x86_64 architecture, your jobs will only run there. If your gatekeeper is x86 (i686 "INTEL"), your jobs would only run there. You have to comment out the line in condor.pm that looks like this to override this feature: 
#    $requirements .= " && Arch == \"" . $description->condor_arch() . "\" ";

Post-Install

OSG 1.2 has introduced a new post-install step that might confuse experienced admins: 
vdt-post-install
This must be done after all Pacman installations have finished.

The VDT install will include a README file located in $VDT_LOCATION/post-install/README file. The OSG install guides will cover all the steps mentioned there; there is no need to follow it right now.
