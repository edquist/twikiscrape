Configuring GRAM Auditing

edit this section

The configuration required to enable and activate GRAM auditing on a Compute Element (CE) is performed by the VDT provided configure_gram_auditing script. This section of the documentation will describe the changes that are effected by the script. The script can be used to disable GRAM auditing, as well.

For general information on the GRAM Auditing, refer to these Globus GRAM documents:

  • GRAM2 (pre-web services):
    http://www-unix.globus.org/toolkit/docs/4.0/execution/prewsgram/Pre_WS_GRAM_Audit_Logging.html
  • GRAM4 (web services):
    http://www.globus.org/toolkit/docs/4.0/execution/wsgram/WS_GRAM_Audit_Logging.html

In order to effect the changes made by the configure_gram_auditing script, the globus_ws init service will be restarted automatically.

Configuring the Database

The default Globus database for the GRAM auditing is Postgres. Since a MySql database is already being used by WS-GRAM (rft_database), MySql will be used rather than install another DBMS on the CE node. The configure_gram_auditing script will create the database based on this information:
  • Database name = auditDatabase
  • Database table = gram_audit_table
  • Database location = $VDT_LOCATION/vdt-app-data/mysql/var
  • Schema location = $VDT_LOCATION/globus/share/gram-service/gram_audit_schema_mysql.sql
  • Database user = Globus user as the VDT determines it.

Two configuration files are updated for the attributes necessary to access the database:

  • GRAM2 (pre-web services): $VDT_LOCATION/globus/etc/globus-job-manager-audit.conf 
   Changing....
         DRIVERCLASS:org.postgresql.Driver
         USERNAME:root
         PASSWORD:auditdbpassword
         URL:jdbc:postgresql://localhost/auditDatabase
   ... to ...
         DRIVERCLASS:com.mysql.jdbc.Driver
         USERNAME:<GLOBUS_USER>
         PASSWORD:<VDT_CREATED_PASSWORD>
         URL:jdbc:mysql://localhost:49151/auditDatabase

  • GRAM4 (web services): $VDT_LOCATION/globus/etc/gram-service/jndi-config.xml
    The "red" lines below show the original values that are replaced. 
     <resource name="auditDatabaseConfiguration"
                 type="org.globus.exec.service.utils.AuditDatabaseConfiguration">
        <resourceParams>
           <parameter>
              <name>factory</name>
              <value>org.globus.wsrf.jndi.BeanFactory</value>
           </parameter>
           <parameter>
              <name>driverClass</name>
              <value>com.mysql.jdbc.Driver</value>
               <value>org.postgresql.Driver</value>
            </parameter>
            <parameter>
                 <name>url</name>
                 <value>jdbc:mysql://localhost:49151/auditDatabase</value>
                 <value>jdbc:postgresql://gratiax31.fnal.gov/auditDatabase</value>
            </parameter>
            <parameter>
               <name>user</name>
               <value>GLOBUS_USER</value>
               <value>root</value>
            </parameter>
            <parameter>
               <name>password</name>
               <value>VDT_CREATED_PASSWORD</value>
               <value>auditdbpassword</value>
            </parameter>
            <parameter>
               <name>globusVersion</name>
               <value>4.0.5 </value>
          </parameter>
      </resourceParams>
    </resource>

Configuring GRAM4 (web services)

Updates to the auditDatabase table in GRAM4 is performed by web services. To activate GRAM auditing for GRAM4, these lines are added to the $VDT_LOCATION/globus/container-log4j.properties file:
        log4j.appender.AUDIT.layout=org.apache.log4j.PatternLayout
        log4j.appender.AUDIT=org.globus.exec.utils.audit.AuditDatabaseAppender
        log4j.additivity.org.globus.exec.service.exec.StateMachine.audit=false
        log4j.category.org.globus.exec.service.exec.StateMachine.audit=DEBUG, AUDIT

This line is added to the above file to turn on logging in the $VDT_LOCATION/globus/var/container-real.log file:

        log4j.category.org.globus.exec.utils.audit=DEBUG

Configuring GRAM2 (pre-web services)

Updates to the auditDatabase table in GRAM2 are performed by a cron service. When a GRAM2 job completes, a single audit file is written to a directory which the cron processes reads from and updates the database.

The configuration script will create this directory if it does not already exist:

   $VDT_LOCATION/vdt-app-data/globus/gram-audit-log

This directory is also known to the cron service and GRAM2 via the $VDT_LOCATION/globus/etc/globus-job-manager.conf file that the script updates with this attribute:

   -audit-directory <YOUR_VDT_LOCATION>/vdt-app-data/globus/gram-audit-log

The cron service is registered, but not enabled by default, as: 

   03,13,23,33,43,53  . <YOUR_VDT_LOCATION>/setup.sh; <YOUR_VDT_LOCATION>/globus/libexec/globus-gram-audit --check

Disabling GRAM Auditing

The configure_gram_auditing script can be used to disable GRAM Trash/Auditing by using the --disable argument. This will not reverse all the changes madae when GRAM auditing was configured, but it will stop the audit log data from being collected.

Disabling GRAM4 (web services)

In the $VDT_LOCATION/globus/container-log4j.properties file, the script will remove all the entries related to GRAM auditing.

Disabling GRAM2 (pre-web services)

In the $VDT_LOCATION/globus/etc/globus-job-manager.conf file, the --audit-directory line will be removed. The cron service will be removed from the vdt-control services file.

Preserving Data From a Previous Installation

When migrating to a new OSG version, the installation is normally performed in a separate directory. This allows for fail-over back to the old version in the event problems occur. However, the new installation needs the following data available that was collected in the old version:
  • the Mysql auditDatabase
  • any GRAM2 audit records that have not been processed in the $VDT_LOCATION/vdt-app-data/globus/gram-audit-log directory.

This migration is triggered by the exporting of the OLD_VDT_LOCATION environmental variable prior to starting the installation.
