Main Oracle JDK 6 Deprecation Ticket (SOFTWARE-890)

Background

Currently, several of our packages are Java-based, and we build them against the Oracle JDK 6. Oracle is officially discontinuing public support for Java 6 at the end of February 2013, including security fixes. This is a problem for us due to the numerous security flaws in Java that people keep discovering. Therefore we must move away from Oracle JDK 6.

Our replacement for Oracle JDK 6 is OpenJDK 7. See the following ticket for discussion: Oracle JDK 6 Replacements Ticket (SOFTWARE-892)

Upgrade instructions

NOTES: java uses the "alternatives" system, so in theory it is possible to install different versions of the jdk side-by-side and pick which one to use. In practice, not so much. The package jdk itself does not use alternatives and instead creates a symlink /usr/bin/java -> /usr/java/default/bin/java, which alternatives is not going to overwrite. It's actually java-1.6.0-sun-compat that provides the alternatives support for oracle jdk. But that gets installed after jdk, at which point the /usr/bin/java symlink is already screwed up.

On the flip side, jdk isn't going to overwrite the /usr/bin/java symlink if it already exists, so if some other version of java is installed first, then the symlink will point to /etc/alternatives/java, and once java-1.6.0-sun-compat is installed, the user can use alternatives --config java to pick which runtime they want to use.

Since we can't be sure which situation the user is going to be in, we can provide the following instructions which, while clunky, should work for either the "installed oracle jdk first" or the "installed other jdk first" situation:

  1. One of these two:
    1. yum install java-1.7.0-openjdk java-1.7.0-openjdk-devel
      • If yum does not find those packages, STOP and update your distro
    2. (once we've switched our packages to require openjdk7) yum update --enablerepo=osg-upcoming
  2. yum erase jdk java-1.6.0-sun-compat (nothing else should get uninstalled by this)
  3. yum reinstall java-1.7.0-openjdk java-1.7.0-openjdk-devel
  4. yum install jdk java-1.6.0-sun-compat
  5. alternatives --config java and pick one of them
  6. alternatives --config javac and pick the same one
  7. Run java -version and javac -version to make sure the right versions are symlinked. They should both be 1.7.something
  8. Restart java services, for example tomcat. See if they work
  9. If everything works, yum erase jdk java-1.6.0-sun-compat

Once we're done with fixing our packages, here's the situations that need to be tested:

  • ( ) Fresh install of osg, with java not already on the system
  • ( ) Fresh install of osg, with oracle jdk on the system
  • ( ) Fresh install of osg, with some other jdk on the system

Rebuilding notes and general wisdom

Virtual dependency for JRE 7 / JDK 7

The virtual dependency provided by java-1.7.0-openjdk (which contains the Java Runtime) is not java, but java7. Likewise, the virtual dependency provided by java-1.7.0-openjdk-devel (which contains the Development Kit) is not java-devel, but java7-devel.

Also note that the Java packages have an epoch of 1.

This means that not only will Oracle JDK 6 falsely match the requirement java-devel >= 1.7.0, but OpenJDK 7 will not match that requirement. Either use java7-devel or java7-devel >= 1:1.7.0.

The jdk, java-sun, java-devel-sun, and java-1.6.0-sun-compat requirements should be replaced by java7-devel. The java requirement should be replaced by java7.

The Fedora packaging guidelines (which we try to follow when we can) for Java state that all Java builds should both Require and BuildRequire jpackage-utils.

Testing

Testing Instructions for OpenJDK 7

The following instructions show how to test java software on openjdk 1.7.0, making sure that no other jdks are installed.

You should do this on a clean VM that you do not plan to reuse for other kinds of tests.

In the first few steps, you will install all the software you need to test:

1 Follow the YumRepositories docs to set up yum repositories.

2 Update osg-release to 3.0-22 or newer to get the osg-upcoming repos 

yum update -y --enablerepo=osg-minefield osg-release

3 Install yum-utils to get yumdownloader 

yum install -y yum-utils

4 Install packages to test from the osg-upcoming repos. (Pass --enablerepo=osg-upcoming-minefield to yum)

5 Ensure OpenJDK 7 is installed 

yum install java-1.7.0-openjdk java-1.7.0-openjdk-devel

In the next set of steps, you will erase the old JDKs to keep them from being used, while not letting yum erase the packages that "require" the old JDKs. This is done by forcefully erasing the packages and then adding entries for them back into rpm's database. That will also allow you to install other Java-based software later without worrying about it bringing in old JDKs.

6 Make a directory to hold RPMs of the old JDKs

mkdir -p /tmp/javarpms; cd /tmp/javarpms

7 Save a list of old jdks that are installed

rpm -qa | egrep '^(java|jdk)-1.[1-6]' | grep -v java-1.7.0 > oldjava.txt

8 Download RPMs for the old JDKs

xargs -a oldjava.txt yumdownloader

9 Erase old jdks, without removing anything that "requires" them

xargs -a oldjava.txt rpm -ev --nodeps

10 Add entries for the old JDKs into rpm's database -- essentially making it pretend they are installed.

rpm -ivh --justdb *.rpm

11 Reinstall openjdk 7 to fix alternatives symlinks, and ensure alternatives are correctly set up 

yum reinstall java-1.7.0-openjdk java-1.7.0-openjdk-devel
alternatives --config java                # the only choice should be openjdk
alternatives --config javac
ls -l /usr/bin/java* | fgrep alternatives # should not be empty
java -version | fgrep 1.7.0               # should not be empty

12 Run your tests. If you want to install more software to test, go ahead; the rpm --justdb line should ensure that no more old JDKs are brought in

Note: last time I tried this, bestman broke because /usr/java/latest didn't exist. Only the jdk package provides that. We might have to patch bestman to not need it? (Also, bestman2 failed to start but test_05_bestman was still successful... might need to look into that)

Note: on el5, jpackage-utils does not create a /usr/share/java-1.7.0 and a /usr/lib/java-1.7.0 directory. Keep an eye out for any problems caused by that; but we want to avoid packaging jpackage-utils if we can.
